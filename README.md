## Overview

This project implements a Software-Defined Vehicle (SDV) data pipeline using Eclipse SDV components. Vehicle sensor data is simulated, processed by Eclipse Kuksa, transported via Eclipse Zenoh, persisted in an Eclipse Ditto digital twin, and monitored via a live terminal dashboard.
```
Python OBD Generator
      │ gRPC
      ▼
Eclipse Kuksa Databroker
      │ gRPC subscribe
      ▼
Bridge (scripts/bridge.py)
      │ Zenoh pub/sub  +  HTTP REST
      ▼              ▼
Eclipse Zenoh    Eclipse Ditto (Digital Twin)
                      │ HTTP REST
                      ▼
               Monitor (scripts/monitor.py)
```

### Signals Published

| Signal | VSS Path | Range |
|--------|----------|-------|
| Vehicle Speed | `Vehicle.Speed` | 0–255 km/h |
| Engine Speed | `Vehicle.OBD.EngineSpeed` | 600–8000 RPM |
| Throttle Position | `Vehicle.OBD.ThrottlePosition` | 0–100 % |
| Coolant Temperature | `Vehicle.OBD.CoolantTemperature` | 70–120 °C |
| Battery State of Charge *(functional modification)* | `Vehicle.Powertrain.TractionBattery.StateOfCharge.Current` | 0–100 % |

## Installation

### 1. Clone the repository
```
git clone https://github.com/hannah-albi/Iteration1-Baseline-SDV-Pipeline-Implementation.git
cd Iteration1-Baseline-SDV-Pipeline-Implementation
```

### 2. Install Python dependencies
```
pip install kuksa-client httpx eclipse-zenoh==0.11.0
```

### 3. Start all Docker containers
```
docker compose up -d
```

This starts: Kuksa Databroker, MongoDB, Ditto (policies, things, things-search, connectivity, gateway), Zenoh router, and Ditto UI.

First startup takes about 2 minutes for Ditto to fully initialize.

### 4. Create the Ditto Thing

Run this once after first startup:
```
curl -X PUT http://localhost:8080/api/2/things/org.vehicle:my-device -H "x-ditto-pre-authenticated: nginx:ditto" -H "Content-Type: application/json" -d "{\"features\":{\"VehicleSpeed\":{\"properties\":{\"value\":0}},\"EngineSpeed\":{\"properties\":{\"value\":0}},\"ThrottlePosition\":{\"properties\":{\"value\":0}},\"CoolantTemperature\":{\"properties\":{\"value\":0}},\"BatteryStateOfCharge\":{\"properties\":{\"value\":100}}}}"
```

## Functional Modification

**Battery State of Charge** was added as a new VSS signal (`Vehicle.Powertrain.TractionBattery.StateOfCharge.Current`). It is generated by `obd_generator.py`, stored in Kuksa, routed through Zenoh by `bridge.py`, persisted in Ditto as the `BatteryStateOfCharge` feature, and monitored with a warning threshold at <25% and critical at <10% in `monitor.py`.

## Iteration 2: System Extension, Validation & Evaluation

### System Extension

In Iteration 2, the SDV pipeline was extended by introducing a message delay in the bridge component (commuincation layer). A 5 second delay was applied to the `VehicleSpeed` signal within the layer.

---

### Functional Validation

The system continues to operate correctly after the modification. All vehicle signals were still successfully recived (from Kuska), published (through Zenoh), and updated (in Eclipse Ditto).

The digital twin remains synchronized with incoming data, with the only visible change being the 5 second delay in the updates for `VehicleSpeed` signal.

---

### Non-Functional Testing

The latency of signal propagation was measured within the communication layer (bridge.py) by recording the timestamps of before and after forwarding from the bridge to the digital twin.

The calculation is as follows:

latency = recieve_time - send_time

Two conditions were evaluated:
1. Normal signals (no delay)
2. Delayed signals (`VehicleSpeed` with a 5-second delay

## Troubleshooting

**Ditto returns 401** — Wait 60–90 seconds after `docker compose up -d` for the cluster to fully form.

**Python module not found** — Make sure you ran `pip install` using the correct Python (not MySQL's Python).

**Port already in use** — Run `docker compose down` then `docker compose up -d` again.
